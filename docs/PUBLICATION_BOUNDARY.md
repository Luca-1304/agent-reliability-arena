# Publication boundary

This document defines what may appear in the compact public showcase for Agent Reliability Arena.

## Public showcase material

The showcase may include:

- the engineering question and controlled comparison design;
- deterministic fixture outcomes labelled as software-validation evidence;
- provider-free orchestration, ledger, repeated-experiment and disclosure demonstrations;
- architecture diagrams and bounded trace examples;
- aggregate counts reconstructed from committed public evidence;
- reproducible commands that make no external request;
- methodology, threat model, limitations and negative outcomes;
- the exact source repository and publication-manifest digest;
- Luca Panayiotou's authorship, problem framing and acceptance standard;
- transparent attribution of AI-assisted implementation, documentation and testing.

Every metric must retain its evidence classification. Failed and aborted outcomes must not be hidden to make the project look stronger.

## Private operational material

The showcase excludes operational material that is useful for running or extending private experiments, including:

- authentication secrets or session material;
- complete private prompts, role inputs and unredacted model outputs;
- provider transaction metadata and raw service payloads;
- private ledgers, operator notes, debugging records and internal logs;
- machine-local paths, host identifiers and environment details;
- enabled execution policies, exact private spending ceilings and retained run roots;
- unpublished ACE control patterns or internal playbooks;
- third-party information that has not been deliberately cleared for publication.

The public package demonstrates the evidence model without handing out the private machinery used to operate it.

## Publication process

1. Build only the files named in `showcase/publication-manifest.json`.
2. Verify every file digest and reconstruct the fixture metrics from the committed public data.
3. Scan the exact bundle for secret-shaped material, machine-local paths, private evidence markers, raw provider metadata, internal notes and unsupported claims.
4. Keep `provider_called: false` and `comparative_claim_permitted: false` for this release.
5. Record any later external publication URL, source commit, date and disclosed limitations.
6. Do not publish a changed bundle until the verifier passes again.

## Claims boundary

This showcase demonstrates deterministic fairness controls, fail-closed parsing, independent verification, tamper-evident evidence, guarded live boundaries, repeated-experiment infrastructure and disclosure-safe publication.

It does not establish hosted-model performance, general superiority of one orchestration pattern, real-world cost efficiency, safety for unrestricted tools, unattended deployment readiness or successful paid-provider execution.
